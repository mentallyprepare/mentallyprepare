# Mentally Prepare Webapp Walkthrough

This folder contains a normal, unzipped copy of the source code needed to build and run the webapp.

## What this app is

Mentally Prepare is a Node.js + Express webapp for an anonymous 21-day emotional journal experience.

A user can:
- join the waitlist
- register and log in
- take an 11-question emotional scan
- get matched with a complementary anonymous partner from a different college
- write one entry per day for 21 days
- read the partner's previous entries with a one-day delay
- comment on older entries
- switch partners if the partner stays inactive long enough
- optionally reveal identities on Day 21 if both people say yes
- export or delete their own data
- receive email and push reminders

There is also:
- an admin dashboard
- waitlist management
- payment hooks for Razorpay and Stripe
- SendGrid email support
- PWA support with a service worker and web manifest

## How the app is assembled

The easiest way to understand the build is to read it in the same order the app boots.

### Step 1: Start the backend server

The whole backend starts in `source/server.js`.

This file does a lot of work:
1. sets global startup error handlers
2. figures out whether the app is in production
3. chooses the SQLite database location
4. opens the database
5. creates tables and indexes if they do not exist yet
6. applies a small schema migration for the waitlist table
7. configures security middleware and JSON parsing
8. serves static frontend files from `source/public/`
9. configures session storage using SQLite
10. adds auth and API rate limiters
11. defines matching, safety, mood-insight, and helper logic
12. configures push notifications
13. configures optional payment providers
14. mounts all route modules
15. schedules daily push reminders
16. exposes health and readiness endpoints
17. starts listening on the configured port

So this app is not split into a separate backend framework and ORM layer. The project keeps the main orchestration inside `server.js`, then pushes route groups into `routes/`.

## Database design

The schema is created directly inside `source/server.js` using SQL statements.

Main tables:
- `users` - account, profile, scan result, consent state, activity, push subscription
- `matches` - pairings between two users
- `entries` - daily journal entries
- `reveals` - Day 21 reveal decisions
- `comments` - comments on older entries
- `reports` - user reports for moderation
- `deletion_log` - anonymized deletion records
- `payments` - payment provider records and status
- `waitlist` - people waiting to join the app

This means the app can spin up on a fresh machine without a manual database migration tool. As soon as `server.js` runs, it creates what it needs.

## Security and middleware

After database setup, `server.js` adds the shared middleware layer.

### Helmet
The app uses Helmet to set security headers and a custom Content Security Policy. The policy allows:
- local scripts and inline scripts
- Google Fonts
- Razorpay checkout scripts and frames
- local images and data URLs

### JSON body parsing
The server accepts JSON request bodies up to 16 KB.

### Static file serving
The frontend files in `source/public/` are exposed through Express static serving.

### Sessions
Authentication is session-based, not JWT-based.

Important details:
- session data is stored in SQLite
- cookies are `httpOnly`
- cookies become `secure` in production
- the session lasts 7 days
- the session secret comes from `SESSION_SECRET` or is generated and saved locally

### Rate limiting
There are two main rate limiters:
- auth limiter for register/login/reset routes
- general API limiter for regular app APIs

## Route breakdown

### 1. Auth routes - `source/routes/auth.js`

This file handles:
- `POST /api/register`
- `POST /api/login`
- `POST /api/logout`
- `POST /api/forgot-password`
- `POST /api/reset-password`

How registration works:
1. validate required fields
2. validate privacy consent
3. validate email format
4. validate password length
5. check for an existing user
6. hash the password with bcrypt
7. insert the user into SQLite
8. create a logged-in session immediately

How login works:
1. find user by email
2. compare password with bcrypt
3. set `req.session.userId`
4. optionally send a welcome-back email

Password reset is currently memory-based for tokens. That means reset tokens live in a `Map` inside the server process and will be lost if the server restarts.

### 2. Core app routes - `source/routes/app.js`

This file powers the main product.

Key endpoints:
- `GET /api/me` - loads the user's full app state
- `POST /api/scan` - saves scan scores and archetype, then tries to match the user
- `POST /api/entry` - writes the day's journal entry
- `GET /api/partner-status` - checks partner activity and switch eligibility
- `POST /api/switch-partner` - leaves an inactive partner and tries to rematch
- `POST /api/comment` - comments on a sealed older entry
- `POST /api/report` - reports a problem
- `POST /api/reveal` - saves the Day 21 reveal choice
- dev-only helper endpoints for test setup and time travel
- `GET /api/my-data` - exports personal data
- `DELETE /api/account` - deletes the user's account and related data
- consent endpoints
- push notification subscription endpoints

The most important route here is `GET /api/me` because it composes almost the entire app state in one response:
- safe user profile data
- match metadata
- the current day
- the prompt for the day
- the user's entries
- partner entries that are already unsealed
- streak count
- reveal status
- comments
- adaptive prompt suggestion
- mood insights
- waiting-state info if the user is not matched yet

That route is basically the frontend state loader.

### 3. Payment routes - `source/routes/payments.js`

This file supports two payment providers.

Razorpay flow:
1. user chooses a product
2. server creates an order with Razorpay
3. server stores a local payment record
4. frontend pays using Razorpay checkout
5. backend verifies the payment signature
6. backend updates local payment status to `paid`

Stripe flow:
1. user chooses a product
2. server creates a checkout session
3. server stores a local payment record
4. user is redirected to Stripe Checkout
5. Stripe webhook confirms completion
6. backend updates local payment status to `paid`

The app keeps payment history in SQLite through the `payments` table.

### 4. Waitlist routes - `source/routes/waitlist.js`

This file supports pre-launch or limited-access growth.

Endpoints:
- `GET /api/waitlist/count`
- `POST /api/waitlist`
- `GET /admin/waitlist`

The waitlist signup flow:
1. validate email
2. check for an existing waitlist row
3. insert a new waitlist row if needed
4. calculate the user's position
5. send a confirmation email if email delivery is configured

### 5. Admin routes - `source/routes/admin.js`

This is the internal operations layer.

Capabilities include:
- opening the admin page
- viewing reports
- viewing users
- viewing app stats
- viewing recent activity
- manual matching of users
- inviting waitlist users
- removing users
- ending matches
- dismissing reports
- exporting admin data

Admin access is protected by a shared password from `ADMIN_PASSWORD`, sent in request headers or a query string.

### 6. Static routes - `source/routes/static.js`

This route file serves:
- privacy page
- terms page
- waitlist page
- app page
- sitemap.xml
- robots.txt

### 7. Waiting entry route - `source/routes/waiting-entry.js`

This is a clever feature for users who have completed onboarding but do not have a partner yet.

It lets them write their Day 1 entry before matching happens.

## Product logic inside `server.js`

Some of the most important product behavior is still implemented directly in `server.js`.

### Prompt system
`prompts` is a 21-item array. Each day maps to one emotional writing prompt.

### Safety scanning
`scanForSafety()` checks entries for:
- crisis language
- self-harm language
- phone numbers
- social handles
- common social media references

If a crisis signal appears, the API returns helpline information.
If possible personally identifying information appears, the frontend can warn the user.

### Adaptive prompts
The app also tries to detect emotional themes from recent writing, such as:
- isolation
- family
- self worth
- fear
- hope
- anger
- grief
- connection
- pressure

If enough entries exist, `getAdaptivePrompt()` suggests a more personalized prompt based on those themes.

### Mood insights
`getMoodInsights()` turns mood selections into a lightweight trend summary, including:
- dominant mood
- whether mood is rising, dipping, or steady
- total word count
- average words per entry
- number of unique moods used

### Matching logic
The matching model is intentionally simple and easy to follow.

Archetype pairs:
- protector <-> connector
- performer <-> disconnector

`attemptMatch(userId)`:
1. loads the current user
2. finds the complementary archetype
3. searches for unmatched users from different colleges
4. applies gender preference filtering
5. applies year preference filtering
6. uses the first valid candidate
7. inserts a match row if a candidate exists

This is a straightforward first-come, first-matched system backed by SQL queries.

### Match day calculation
`getMatchDay(startedAt)` computes the day number using UTC day boundaries and clamps the result to 1 through 21.

### Account and data deletion
`deleteUserDataTx` runs inside a SQLite transaction and removes:
- matches tied to the user
- entries
- reveals
- comments
- reports
- payments
- the user record
- plus an anonymized row in `deletion_log`

## Push notification system

The app supports web push notifications.

How it works:
1. on startup, the server loads or generates VAPID keys
2. the frontend registers `source/public/sw.js`
3. the frontend requests notification permission
4. the frontend subscribes through the browser push manager
5. the backend stores the subscription JSON on the user row
6. the backend can later send push payloads using `web-push`

The scheduled reminder system in `server.js` aims to send daily push notifications around 8 PM IST.

## Email system

- `source/lib/sendgrid.js` - low-level SendGrid sender
- `source/lib/email.js` - higher-level app email helpers

- `source/email-service.js` - pass-through wrapper used by the server

Emails currently cover:

## How to run this exported source locally

1. open a terminal inside `source/`
2. create a `.env` file from `.env.example`
3. fill in your secrets
4. run `npm install`
5. run `npm start`
6. open `http://localhost:8080`

Because the schema is created in code, a fresh SQLite database file will be created automatically when the app starts.

## If you wanted to rebuild this app from scratch

Here is the clean mental model:

1. create an Express server
2. connect SQLite using `better-sqlite3`
3. create the tables for users, matches, entries, reveals, comments, reports, payments, and waitlist
4. add session auth using `express-session` and `connect-sqlite3`
5. add registration and login routes
6. build the 11-question scan on the frontend
7. map scan scores to an archetype
8. implement complementary-user matching rules
9. create daily prompt logic for 21 days
10. create entry writing, delayed reveal of partner entries, and commenting
11. add inactivity checks and rematching logic
12. add Day 21 consent-based reveal logic
13. add admin tooling for moderation and manual matching
14. add SendGrid emails and web-push reminders
15. add PWA manifest, service worker, and deployment configs
16. polish the landing page and app screens with CSS and animations

That is exactly the order this codebase roughly follows.
